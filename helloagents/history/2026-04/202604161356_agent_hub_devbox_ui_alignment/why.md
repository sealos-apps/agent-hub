# 变更提案: Agent Hub 对齐 DevBox 产品风格重构

## 需求背景
当前 Agent Hub 已经具备创建实例、列表管理、聊天、终端、文件管理、AIProxy 自动注入、Hermes 模板部署等核心能力，但前端承载方式仍以单页面和多个 Modal 为主，存在以下问题：

- 首页承担了过多职责，列表、创建、编辑、聊天、终端、文件管理都在同一页面上下文中完成，信息密度过高。
- 页面结构与 Sealos DevBox 不一致，用户在同一产品体系内切换时需要重新学习交互模式。
- Agent 的详情能力没有独立工作台，聊天、终端、文件等核心能力都通过弹窗进入，缺少稳定的长驻信息架构。
- 现有视觉风格虽然已做卡片化，但与 DevBox 的页面留白、Header 结构、按钮体系、状态标签、详情页布局仍不统一。

本次变更的目标不是重写业务能力，而是基于当前功能，参考本地源码 `reference/sealos/frontend/providers/devbox` 的页面组织与 UI 风格，对 Agent Hub 做一次产品级前端重构，使其在同一产品线下具备几乎一致的体验语言。

## 产品分析

### 目标用户与场景
- **用户群体:** 在 Sealos 内同时使用 DevBox 与 Agent Hub 的开发者、AI 工程师、自动化用户。
- **使用场景:** 用户在 Sealos 工作区中创建 Agent、配置模型、查看启动状态、进入聊天、终端与文件管理。
- **核心痛点:** 同一产品体系下交互模式不一致；首页信息过载；高级能力入口分散；实例详情缺少稳定的工作台。

### 价值主张与成功指标
- **价值主张:** 让 Agent Hub 成为与 DevBox 同风格、同层级、同交互范式的 Sealos 产品模块，同时保持 Agent 业务的专属能力表达。
- **成功指标:**
  - 新用户可在 3 步内完成“进入列表页 -> 创建实例 -> 进入详情页”。
  - 首页不再显示与 Agent 核心操作弱相关的信息卡片和 API 地址。
  - 聊天、终端、文件管理具备稳定的详情页入口，不再仅依赖弹窗。
  - 视觉上与 DevBox 的 Header、列表、状态标签、详情页结构保持同一产品感知。

### 人文关怀
- 保持信息层级清晰，减少用户在创建和排查问题时的认知负担。
- 将敏感信息继续留在系统自动注入链路中，不在主界面直接回显密钥。
- 通过统一的状态表达与错误反馈，降低用户对“实例是否真的失败”的误判概率。

## 变更内容
1. 将 Agent Hub 从单页面结构重构为“列表页 / 创建页 / 详情页”三段式产品结构。
2. 参考 DevBox 的页面骨架与组件模式，统一 Header、列表、按钮、状态标签、详情页导航和内容区布局。
3. 保留当前聊天、终端、文件、模型配置、AIProxy 注入等功能，但调整其挂载位置与交互路径。
4. 拆分前端页面、表格列、状态与操作逻辑，降低单文件复杂度，便于后续继续扩展多 Agent 模板。

## 影响范围
- **模块:** 前端路由、Agent 列表页、创建流程、详情页、聊天/终端/文件入口、视觉样式、状态轮询逻辑。
- **文件:** `web/src/App.tsx`、`web/src/app/pages/AgentHubPage.tsx`、`web/src/components/business/agents/*`、`web/src/app/pages/agent-hub/hooks/*`、`web/src/index.css`、新增路由与详情页相关文件。
- **API:** 复用现有 `/api/v1/agents`、`/api/v1/agents/:agentName`、聊天/终端/文件相关接口，前端调用路径会调整。
- **数据:** 不引入新的后端持久化数据模型，主要新增前端页面状态和视图模型组织方式。

## 核心场景

### 需求: Agent 列表页与 DevBox 首页风格对齐
**模块:** Agent 列表页
用户进入 Agent Hub 首页时，应获得与 DevBox 首页接近的产品结构和视觉语言。

#### 场景: 浏览实例列表
用户打开 Agent Hub 首页。
- 顶部显示统一的页面 Header、搜索与创建入口。
- 中间区域以列表形式展示实例，不再显示 API 地址、Backend Managed 卡片、当前能力卡片。
- 列表行展示别名为主、`agentName` 为辅、状态、资源规格与快捷操作。

### 需求: 创建流程页面化
**模块:** Agent 创建页
用户创建 Agent 时，应进入与 DevBox 创建页同层级的完整页面，而不是承载越来越重的 Modal。

#### 场景: 创建新实例
用户点击创建按钮。
- 进入独立创建页，而不是停留在列表页内层层弹窗。
- 页面中完成模板选择、别名填写、资源规格选择、模型选择和 AIProxy 自动注入。
- 创建完成后进入对应 Agent 详情页。

### 需求: 提供 Agent 详情工作台
**模块:** Agent 详情页
Agent 的核心能力需要稳定的详情工作台承载，而不是由首页弹窗临时挂载。

#### 场景: 进入实例详情
用户从列表页点击某个实例。
- 进入详情页，页头显示返回、名称、状态和主要操作。
- 左侧为简洁的 tab 导航。
- 右侧根据 tab 展示 Overview、Chat、Terminal、Files、Deployment 等内容。

### 需求: 保留现有能力并迁移到新结构
**模块:** 聊天、终端、文件管理
在不重写底层协议的前提下，将当前能力迁移到新的产品结构中。

#### 场景: 使用聊天、终端和文件
用户在详情页点击对应 tab。
- 聊天、终端、文件功能继续可用。
- 原有 hook 逻辑复用，优先迁移挂载方式，不先推倒重写协议。

### 需求: 统一状态与刷新感知
**模块:** 状态轮询与错误反馈
用户应能清楚理解实例真实状态，而不是被冗余 warning 或页面滞后状态误导。

#### 场景: 实例从创建中进入可用
用户创建实例后返回列表页或进入详情页。
- 前端持续轮询直到状态从 creating 进入 running/ready。
- 页面状态与后端 `/api/v1/agents` 返回保持一致。
- 错误和 bootstrap 信息集中展示，不与视觉卡片混杂。

## 风险评估
- **风险:** 一次性同时引入路由、页面重构和视觉重构，容易造成功能回归。
- **缓解:** 采用分阶段实施，优先引入页面骨架与路由，再逐步迁移列表、创建、详情和细节能力。
- **风险:** 直接复制 DevBox 的基础设施展示会削弱 Agent Hub 的产品定位。
- **缓解:** 仅复用 DevBox 的页面骨架、视觉语言和交互模式，不复制其运维信息密度。
- **风险:** 聊天、终端、文件 hook 当前体量较大，迁移时容易引入状态同步问题。
- **缓解:** 先迁移挂载位置和页面结构，不立即重写底层通信协议。
