## 产品说明

Agent Hub 是一个运行在 Sealos 内部 App / Launchpad 体系中的 Agent 实例部署与运行控制台。它的目标不是让用户“自己发明一个 Agent”，而是让用户在 Sealos 工作空间内，以尽量少的容器知识，快速创建并运行热门、适合容器化部署的 Agent 实例。

当前产品方向严格基于参考项目现状抽象：应用启动后通过 Sealos Desktop SDK 获取当前用户会话、语言、配额和宿主域名信息，解析当前 workspace 的集群上下文，然后按统一 label 聚合 Devbox、Service、Ingress 三类资源，以“Agent 实例”的视角展示和操作。用户通过右上角“创建”按钮进入模板选择流程，选择模板后进入配置弹窗，提交后统一创建 Devbox / Service / Ingress；部署完成后，用户可以直接复制 API 地址与 Key，或进入“对话”和“终端”入口开始使用。

产品面向的不是熟悉 Kubernetes 的运维用户，而是希望“热门 Agent 有了就能在 Sealos 里一键跑起来”的普通技术用户、产品用户和 AI 使用者。为了保证丝滑，系统需要保留关键配置项，但必须通过模板默认值、自动生成资源名/域名/API Key、统一创建流程和部署后即时入口，尽量把底层复杂度包住。

首批模板按当前方向锁定为：
- Hermes Agent
- OpenClaw

说明：
- UI 文案可以继续使用“创建 Agent”
- 技术语义统一定义为“创建 Agent 实例”，避免后续开发把它误解成“让用户定义一个新的 Agent 产品”

## 功能需求

- 初始化 Sealos Desktop SDK，并在应用加载时自动获取当前用户的 session、语言、workspace 配额和 host config
- 解析 session 中的 kubeconfig，构建当前 workspace 的 cluster context，包括 server、namespace、token、operator 等信息
- 以 `agent.sealos.io/name` 与 `app` label 为主键，聚合当前用户名下的 Devbox、Service、Ingress 资源，形成“Agent 实例”列表
- 展示 Agent 实例列表，支持搜索、状态查看、更新时间查看、API 地址查看、API Key 复制
- 提供右上角“创建”按钮，点击后弹出 Agent 模板选择弹窗
- 在模板选择弹窗中展示 Hermes Agent 与 OpenClaw 两张模板卡片
- 支持 Hermes Agent 模板完整创建流程
- 预留 OpenClaw 模板的同构创建入口；若模板尚未完成，可先呈现为“后续开放 / Beta”状态，但整体交互框架不得改变
- 根据模板自动生成创建 blueprint，包括实例名、域名前缀、完整域名、API Key、默认镜像、默认端口、默认启动命令、默认资源规格
- 打开配置弹窗时展示模板生成结果与用户可编辑的关键配置
- 保留当前参考版中的关键资源配置项：CPU、内存、存储
- 保留实例名展示；当前参考版保持只读，由系统自动生成，后续如要允许用户命名，需在不影响资源唯一性的前提下单独设计
- 将模板私有的底层参数继续由模板托管，包括但不限于镜像、启动命令、工作目录、协议、端口、自动生成的 API Key、自动生成的域名
- 一次提交统一创建 Devbox、Service、Ingress 三类资源，不让用户分别操作底层 K8s 资源
- 支持统一编辑实例配置，复用创建弹窗
- 支持按实例维度联动删除 Devbox、Service、Ingress，避免用户只删掉其中一个资源造成脏状态
- 为每个实例提供“对话”入口，通过实例当前暴露的 OpenAI-compatible API 地址和 Key 直接发起会话
- 为每个实例提供“终端”入口，通过 Pod exec WebSocket 连入容器，直接操作 Agent CLI 或运行环境
- 在应用内展示提交结果、复制结果、错误信息和连接状态，减少用户离开当前页面查错的成本
- 为不懂容器的用户提供模板默认值和自动生成值，确保大多数情况下用户改完关键配置即可直接部署
- 预留后续 AI 能力增强位：
  - AI 推荐 CPU / 内存 / 存储默认档位
  - AI 分析部署失败信息并给出修复建议
  - AI 根据模板自动解释配置项含义

## UI 布局

### 整体布局

整体采用单页面、全屏、高度占满的控制台布局，结构严格贴近参考版：

- 顶部是固定头部区域
- 中间是唯一主内容区
- 创建、配置、对话、终端全部通过居中的 Overlay / Modal 打开
- 用户看到的是“Agent 实例维度”的界面，不直接暴露 Devbox / Service / Ingress 这类底层资源切换

这点很重要：参考版虽然内部维护了 `devbox / service / ingress` 三类数据，但主界面实际已经收敛为 Agent 列表，不应该把底层资源类型切换暴露给普通用户。

### 顶部 Header

Header 分为左右两部分：

- 左侧：
  - 产品名称 `Agent Hub`
  - 文档按钮
  - 当前 cluster / workspace 上下文提示信息
- 右侧：
  - 搜索框
  - 主按钮 `创建`

控件规范：

- 产品标题放在最左侧，字号最大，是页面唯一一级视觉标题
- 文档按钮紧跟标题，样式弱于创建按钮
- 搜索框靠右，宽度固定，支持按实例名、备注、状态、地址等关键词模糊搜索
- 创建按钮位于最右侧，是页面最高优先级主操作按钮

### 主内容区

主内容区是一个大卡片容器，内部展示 Agent 实例表格。

表格列维持参考版结构：

- 名称
- 负责人
- API 地址
- Key
- 状态
- 更新时间
- 操作

行级操作维持参考版结构：

- 对话
- 终端
- 配置
- 删除

交互规则：

- API 地址以按钮形式展示，点击即复制
- Key 以“复制 Key”按钮形式展示
- 状态使用 badge 表现
- 操作按钮按使用频率排序：对话、终端、配置、删除
- 若资源列表为空，可展示首个实例引导态，但不改变“右上角创建”为主入口的策略

### 创建类型弹窗

点击 Header 右上角“创建”按钮后，弹出模板选择弹窗。

当前弹窗结构沿用参考版：

- 标题：选择创建类型
- 中间是模板卡片区
- 底部是全宽“下一步”按钮

模板卡片规则：

- 每张卡片包含 Logo、模板名称、模板状态说明
- 首批固定展示：
  - Hermes Agent
  - OpenClaw
- 若某模板暂未开放，卡片显示禁用态和“后续开放 / Beta”标记，但位置保留，避免用户误以为产品只支持单模板

### 配置弹窗

模板选择完成后进入配置弹窗。

参考版的当前可见配置项是：

- 应用名（只读）
- CPU
- 内存
- 存储限制

参考版当前隐藏但由模板托管的字段包括：

- namespace
- image
- port
- user
- workingDir
- args
- serviceType
- protocol
- apiKey
- apiUrl
- fullDomain
- domainPrefix

V1 配置弹窗设计规则：

- 第一屏必须直接看到关键配置，不允许把 CPU / 内存 / 存储全塞到高级设置中
- 当前参考版里应用名维持只读，由系统自动生成
- CPU、内存、存储为可编辑主字段
- 弹窗内部必须预留“模板私有字段区”
  - Hermes Agent 在 V1 可暂时没有额外可见私有字段
  - OpenClaw 后续若需要额外参数，可放入该区域，而不推翻整体弹窗结构
- 提交按钮位于弹窗底部，全宽展示，文案为“统一创建资源”或“统一保存资源”

### 对话弹窗

实例点击“对话”后，打开对话弹窗。

布局结构：

- 顶部状态区：
  - 当前 API 地址
  - 当前连接方式（SSE / WebSocket）
  - 当前状态
  - 错误信息
- 中部消息区：
  - 消息列表滚动展示
  - 用户消息右对齐
  - Agent 消息左对齐
  - 支持流式接收中状态
- 底部输入区：
  - 多行输入框
  - 发送按钮

设计要求：

- 用户无需离开 Agent Hub，即可验证实例是否真正能响应
- 这个入口是“部署成功后的第一可用证明”

### 终端弹窗

实例点击“终端”后，打开终端弹窗。

布局结构：

- 顶部状态区：
  - Pod 名称
  - 容器名称
  - 当前连接状态
  - 错误提示
  - 重连 / 断开按钮
- 中部主区：
  - xterm 终端区域，全宽全高

设计要求：

- 终端是兜底入口
- 当对话不可用、服务未完全启动、或需要手动检查 Agent 进程时，用户必须能立刻进入 CLI

## 用户使用流程

### 流程一：打开应用并查看已有实例

1. 用户从 Sealos 内部打开 Agent Hub
2. 应用初始化 Sealos Desktop SDK
3. 应用读取 session、host config、quota 和 kubeconfig
4. 应用构建当前 workspace 的 cluster context
5. 应用从 `/k8s-api` 聚合拉取当前用户名下的 Devbox、Service、Ingress
6. 页面展示实例列表
7. 用户可直接搜索、复制 API、复制 Key、进入对话或终端

### 流程二：创建 Hermes Agent 实例

1. 用户点击右上角“创建”
2. 弹出模板选择弹窗
3. 用户选择 `Hermes Agent`
4. 系统生成默认 blueprint：
   - 自动生成实例名
   - 自动生成域名前缀与完整域名
   - 自动生成 API Key
   - 注入默认镜像、默认端口、默认启动命令
5. 打开配置弹窗
6. 用户确认或修改 CPU、内存、存储等关键配置
7. 用户点击“统一创建资源”
8. 系统统一创建 Devbox、Service、Ingress
9. 系统刷新实例列表
10. 用户从列表中复制 API 地址 / Key，或直接进入“对话”/“终端”

### 流程三：创建 OpenClaw 实例

1. 用户点击右上角“创建”
2. 弹出模板选择弹窗
3. 用户选择 `OpenClaw`
4. 如果模板已开放：
   - 进入与 Hermes 相同的配置与创建框架
   - 使用 OpenClaw 自己的模板 blueprint
   - 统一创建实例资源
5. 如果模板尚未开放：
   - 卡片展示禁用状态
   - 用户可见“后续开放 / Beta”标记
   - 不改变后续版本的总体交互结构

### 流程四：部署后验证实例可用性

1. 用户在实例列表找到刚创建的实例
2. 若需要快速验证，点击“对话”
3. 系统通过当前实例的 OpenAI-compatible API 地址与 Key 建立连接
4. 用户输入测试消息，观察 Agent 是否正常返回结果
5. 若对话失败或需要进一步排查，用户点击“终端”
6. 系统定位 Pod 并建立容器终端连接
7. 用户直接在容器内检查进程、日志、CLI 状态

### 流程五：调整配置或删除实例

1. 用户点击实例行中的“配置”
2. 系统从当前 Devbox / Service / Ingress 回填配置
3. 用户修改 CPU、内存、存储等可编辑项
4. 用户提交后，系统统一更新三类资源
5. 若用户点击“删除”，系统按实例维度联动删除相关资源，避免残留脏对象

## 补充说明

### 实例与底层资源映射

| 概念 | 说明 |
|------|------|
| Agent 实例 | 用户在 Agent Hub 中看到和操作的对象 |
| Devbox | 真正承载 Agent 容器运行环境的核心资源 |
| Service | 为 Agent 容器暴露集群内访问端口 |
| Ingress | 为 Agent 实例分配外部访问地址 |
| API 地址 | 通常由 Ingress 域名拼接生成，用于对话入口 |
| API Key | 创建时自动生成，用于访问实例对外 API |

### 模板支持矩阵

| 模板 | 参考版状态 | 部署后入口 | 说明 |
|------|-----------|-----------|------|
| Hermes Agent | 已实现主流程 | 对话、终端 | 当前参考版的真实可创建模板 |
| OpenClaw | 已有入口卡片，模板细节待完善 | 对话、终端 | 交互入口必须保留，具体 blueprint 可后补 |

### 当前参考版关键约束

| 项 | 当前约束 |
|----|---------|
| 首页入口 | 右上角固定“创建”按钮 |
| 首页视图 | 实例列表，不是模板商城首页 |
| 创建方式 | 先选模板，再填配置 |
| 用户可见资源配置 | CPU、内存、存储 |
| 模板默认值 | 由系统自动生成或模板内置，不直接让用户逐项处理 |
| 部署后核心入口 | 对话、终端 |

### 需要明确保留的产品决策

- UI 文案继续保留“创建 Agent”
- 产品文档和技术实现统一使用“创建 Agent 实例 / 部署 Agent 实例”
- 不对普通用户暴露 Devbox / Service / Ingress 资源类型概念
- 不要求用户理解 Kubernetes，只要求用户理解“我在 Sealos 里创建了一个可用的 Agent”

### 后续增强建议

- 为 CPU / 内存 / 存储提供模板推荐值和推荐说明
- 对部署失败增加 AI 诊断
- 增加模板元数据能力，让每个 Agent 模板自己声明：
  - Logo
  - 名称
  - 默认镜像
  - 默认端口
  - 默认启动命令
  - 是否支持对话
  - 是否支持终端
  - 是否处于 Beta / 未开放状态

## 应用场景

- AI 使用者想在 Sealos 里快速跑起一个 Hermes Agent，不想手写 YAML，也不想自己配 Devbox、Service、Ingress
- 团队成员想验证某个热门 Agent 是否适合当前业务场景，直接在 Sealos 内点“创建”，几分钟内拿到一个可对话、可进入终端的实例
- 产品或运营同学不懂容器，但需要一个可访问的 Agent 地址和一组可复制的凭据，用于内部测试或演示
- 技术同学在部署后发现 Agent 行为异常，先在页面里点“对话”验证，再通过“终端”进入容器排查 CLI 和进程
- 团队准备跟进新的热门 Agent 模板时，只需在现有创建框架中新增模板卡片与 blueprint，而不必推翻整个 Agent Hub 的主界面和运行逻辑
