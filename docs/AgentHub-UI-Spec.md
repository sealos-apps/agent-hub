# Agent Hub UI 规范（v3）

## 0. 规范来源
- 产品上下文：`.impeccable.md`
- 方法论来源：`impeccable` skill（重点采用 `extract` 与 `craft` 的约束）
- 适用代码范围：
  - `web/src/app/pages/agent-hub/**`
  - `web/src/components/business/**`
  - `web/src/components/ui/**`
  - `web/src/index.css`

本规范是 Agent Hub 前端唯一视觉与交互基线。新页面、新组件与重构必须遵循本文件。

---

## 1. 产品定位与设计目标

### 1.1 产品定位
- Agent Hub 是 Sealos 生态内的工作台模块，不是独立营销站点。
- 使用环境以桌面弹窗/中小窗口为主，不以全屏展示优先。
- 用户目标是高频运维操作：创建、查看状态、排障、进入终端/文件/设置。

### 1.2 设计目标
- 与 DevBox 保持同体系语言：系统级、克制、紧凑、可信。
- 优先信息密度与操作效率，不做视觉炫技。
- 全站统一尺度、统一组件 API、统一状态表达。

---

## 2. 核心原则（强制）

### 2.1 一致性优先于局部美化
- 禁止在业务页面手写第二套按钮、输入框、状态样式。
- 任何视觉差异优先通过 `ui` 组件 `variant/size` 扩展实现。

### 2.2 只允许一个主滚动轴
- 每个工作区只允许一个主纵向滚动容器。
- 禁止靠横向滚动条兜底正常信息展示。

### 2.3 密度优先（窗口模式）
- 默认正文字号为 `12px`。
- 标题层级通过字重和间距建立，不依赖大字号。

### 2.4 组件抽取门槛
- 相同意图、同结构、出现 3 次及以上，必须抽取为复用组件或 token。
- 出现 1~2 次的场景禁止过度抽象。

### 2.5 禁止 AI 风格反模式
- 禁止卡片左侧/右侧彩色强调条（`border-left/right > 1px`）。
- 禁止 gradient 文本（`background-clip: text`）。
- 禁止卡片套卡片作为常规布局。
- 禁止“图标 + 标题 + 描述”重复堆砌式卡片墙替代表格信息流。

---

## 3. Design Token 体系（执行级）

## 3.1 Token 分层
- `Primitive Token`：底层色值、半径、阴影、间距。
- `Semantic Token`：业务语义（surface、text-muted、danger 等）。
- 页面与组件仅允许引用 `Semantic Token`，禁止直接写十六进制色值（过渡阶段除外）。

## 3.2 当前颜色语义（以 `web/src/index.css` 为准）
- `--color-brand`: 主操作色
- `--color-brand-hover`: 主操作 hover
- `--color-bg`: 页面底色
- `--color-surface`: 容器底色
- `--color-text`: 主文本
- `--color-muted`: 次级文本
- `--color-border`: 默认边框
- `--color-success`: 成功状态
- `--color-danger`: 危险状态

### 3.2.1 颜色使用规则
- 主文本/正文必须使用 `--color-text` 或其语义派生，不使用硬编码深灰。
- 次级说明统一使用 `--color-muted`。
- 边框统一 `--color-border`，避免页面内出现多套灰阶边框。
- 表单错误仅使用 danger 语义，不混用橙色/紫色等自定义告警色。

## 3.3 字体与字号基线
- 基础字体：`SF Pro Display / PingFang SC / Helvetica Neue / sans-serif`
- 根字号：桌面 `13.5px`，`<=640px` 为 `14px`（`web/src/index.css`）
- UI 组件字号（固定尺寸，不使用 fluid type）：
  - 页面主标题：20px（`text-[1.28rem]`）
  - 区块标题：16px（`text-[1.02rem]`）
  - 组件标题：13px / 12px
  - 正文：12px
  - 辅助说明：11px
  - 状态标签：10px

说明：产品型工作台使用固定字号体系，不使用营销页的 `clamp` 流体排版。

## 3.4 间距基线（4pt）
- 标准间距序列：`4, 8, 12, 16, 24, 32`
- 组件内默认垂直节奏：`space-y-1.5` / `gap-2.5` / `gap-3`
- 同类区块间距必须一致，禁止逐块“手调 margin”。

## 3.5 圆角与阴影
- 控件圆角：`rounded-lg`
- 卡片圆角：`rounded-xl`（默认）/ `rounded-2xl`（强调）
- 阴影仅允许轻阴影，不使用大面积浮层阴影作为强调手段。

## 4. 组件规范（必须复用）

## 4.1 标准组件清单
- `Button`
- `Input`
- `Select`
- `SearchField`
- `Modal`
- `StatusBadge`
- `EmptyState`

## 4.2 Button
- API：
  - `variant`: `primary | secondary | ghost | danger`
  - `size`: `xs | sm | md`
- 尺寸映射：
  - `xs`: `h-7 px-2 text-[10px]`（仅高密度行内动作）
  - `sm`: `h-8 px-2.5 text-[11px]`
  - `md`: `h-9 px-3 text-[12px]`（默认）
- 禁止项：
  - 页面直接通过 `className` 改按钮高度和字号。
  - 同一工具栏无主次地混用 `sm` 与 `md`。

## 4.3 Input / Select / SearchField
- 默认 `size="md"`，与主按钮同高（`h-9`）。
- `label` 与 `hint` 采用固定字级：
  - label: 12px
  - hint: 11px
- 错误文案统一 danger 语义，禁止页面单独定义错误色。

## 4.4 Modal
- 保留工作台视觉，不做营销化弹窗。
- 弹窗内容区必须可滚动，不允许内容溢出截断。
- 表单类 Modal 底部操作栏固定布局：右对齐、同尺寸按钮组。

## 4.5 复用抽取清单（v3 必做）
- 抽取 `WorkspaceHeader`：统一标题、副标题、右侧动作组。
- 抽取 `InfoFieldRow`：统一“字段名 + 值 + 辅助说明”信息展示。
- 抽取 `SectionCard`：统一内容卡结构（header/body/footer）。

## 5. 页面骨架规范

## 5.1 列表页（Agents List）
- 结构：`Header + Filters/Toolbar + Table + EmptyState`
- 顶部动作（搜索、模板市场、创建 Agent）统一 `h-9`。
- 列表项优先表格化，避免重卡片化瀑布布局。

## 5.2 模板市场页（Template Select）
- 模板卡片在中宽及以上视图应支持一行多列。
- 页面主容器宽度与创建页保持一致（同一内容宽度基线）。
- 默认仅显示“全部模板”时，仍保持卡片栅格对齐与密度。

## 5.3 创建页（Create Agent）
- 结构：`Header + Main Form + Sidebar Summary`
- 窄宽下保持左右结构优先，必要时压缩间距再切换层叠。
- 创建按钮状态必须与创建过程状态绑定（创建中禁止重复提交）。

## 5.4 详情页（Agent Detail）
- 固定骨架：`Header + Sidebar + Workspace`
- 禁止改为顶部 tab 横排替代侧栏。
- Sidebar 允许内部滚动；Workspace 为主滚动容器。

## 5.5 文件工作台（Files）
- 单击：选择；双击：目录进入/文件预览（统一交互，不混用）。
- 顶部路径展示与路径输入合并为一行，避免信息重复。
- 目录切换必须保持路径规范化（禁止出现 `//root` 等脏路径）。

---

## 6. 交互与状态规范

## 6.1 状态优先级
- 主状态：`creating / running / paused / failed / deleting`
- 次状态：bootstrap 阶段、连接中、重连中
- 页面只展示对当前动作有帮助的状态，不展示技术噪音字段。

## 6.2 按钮可用性
- 不可执行动作必须禁用并给出原因（`reason`）。
- 创建中/启动中状态下，互斥动作禁止点击。

## 6.3 异步反馈
- 列表轮询与详情恢复逻辑必须避免闪烁和重复渲染。
- 计时器、轮询、websocket 在页面卸载时必须清理。

---

## 7. 文案规范（UX Writing）

## 7.1 文案原则
- 说人话、可执行、短句优先。
- 禁止“系统内部解释型”句子直接暴露给用户。
- 同一概念在全站只允许一个名称（例如“集群内服务地址”不能多种叫法）。

## 7.2 文案禁用示例
- 禁止：状态原文、内部枚举值（如 `Running`、`workspace-aiproxy`）直接展示。
- 禁止：开发者口吻描述（如“这里只展示来源，不展示密钥内容”作为主文案）。

---

## 8. 响应式与窗口模式规则

## 8.1 小窗口优先
- 目标窗口：桌面弹窗中小尺寸优先（非全屏）。
- 优先级：保持可读 > 保持可操作 > 保持完整信息。

## 8.2 空间不足处理顺序
1. 收缩横向间距
2. 次要操作换行
3. 折叠次要信息块
4. 最后才允许结构层叠

## 8.3 禁止项
- 禁止把横向滚动条作为常规布局方案。
- 禁止通过强制大 `min-width` 解决拥挤。

---

## 9. 可访问性基线
- 文本与背景满足最小可读对比（正文至少 4.5:1）。
- 所有可点击控件保留可见焦点态。
- 关键操作可通过键盘触达（Tab 顺序正确）。
- 错误提示必须有可见文本，不仅靠颜色区分。

---

## 10. 开发执行规范（给前端与 AI）

## 10.1 改动顺序
1. 先改 token / 基础组件
2. 再改页面骨架
3. 最后改局部业务样式

## 10.2 代码约束
- 新样式优先进入 `ui` 层，不在页面散落临时 class。
- 一个 PR 内禁止同时引入新视觉体系和新业务逻辑（降低回归风险）。
- 页面新增样式必须可追溯到本规范条款。

## 10.3 合并前检查清单
- 尺寸基线是否统一（按钮、输入、标签）
- 是否出现硬编码色值替代语义 token
- 是否出现禁止项（侧条强调、gradient 文本、卡片套卡片）
- 小窗口是否可完整操作（列表、创建、详情、文件）
- 异步状态是否有明确反馈且无闪烁

---

## 11. 版本策略
- 当前版本：v3
- 若新增组件规范或全局 token，必须更新本文件并在 PR 描述中声明“规范变更点”。
- 未更新规范的视觉改动视为临时实现，不允许长期保留。
